# gigMCP Iteration 01 — Initial Codification Candidates

## Methodology

This is the first iteration in the standing directive set 2026-05-06. The
directive has been live for 7 days with **zero outputs written**; this report
backfills the missed cadence rather than starting cleanly from today.

**Scope discipline (per directive):** 6–10 high-leverage candidates ready to be
codified as MCP tools *today*, not 30. Selection emphasizes stable surfaces
already shipping HTTP endpoints — these are the cheapest to wrap and the most
honest about what they will guarantee.

**Surfaces scanned (≤25 file reads):**

- `decision-engine/` — `api/routes.py`, `engine/certificates.py` (header), `CLAUDE.md`, drift_sentinel reports dir listing
- `decision-engine/drift_sentinel/` — `README.md`, `drift_scan.py` (header + handler index), `DRIFT_RULES.yaml` (presence)
- `user-access-engine/` — `openapi.yaml` (~paths), `api/services/` listing (access.py, google_oauth.py, sessions.py, state_tokens.py)
- `intelligence-silo/` — `CLAUDE.md`, `core/` listing (orchestrator, slm matrix, memory hierarchy, vault, bridge)
- `gigaton-engine/` — `pricing_engine/api.py`, `multi_agent/api.py`
- `sales-operating-system/` — `app/services/` listing, `app/api/routes.py` route inventory, `CLAUDE.md`
- `ppeme/` — `openapi.yaml`, `api/routes/` listing
- `human-management-engine/` — `openapi.yaml`, `api/` listing
- `gigaton-gateway/` — `api/routing_table.py` (full)
- `sovereign-influence-engine/` — `README.md`, `services/` listing, `services/operator-api/src/` listing
- Memory: `gigmcp_recurring_analysis_directive.md`, `tool_integration_registry.md`, canonical first principles framework index

**Ranking heuristic (in priority order):**

1. **Live HTTP endpoint exists** (codification ≈ schema-stable wrapper, not new logic)
2. **Already gateway-routed** (path-prefix in `gigaton-gateway/routing_table.py` means the platform already treats the verb as canonical)
3. **Cited in ≥2 doctrine frameworks** (multi-framework anchorage = stability signal)
4. **Provider-abstraction-safe** (deterministic surface, or LLM call already centralized behind `_invoke_llm`)
5. **Multiplies a human super-ability** (per Gigent Value Matrix audit chain — `functionality → capability → super-ability enabled`)
6. **Low blast-radius if exposed prematurely** (read-mostly or governed-write; not raw state mutation)

Candidates fail the cut if they (a) are pure scaffold with no endpoint yet
(HME and PPEME at v0.1.0-scaffold — endpoints declared in openapi.yaml but
implementation incomplete; deferred to iteration 02–03), (b) require provider
abstraction not yet remediated (none surfaced this iteration), or (c) bypass
RTQL trust qualification at write time.

## Candidates

### 1. `gigaton.decision.process`

- **What:** Run a DecisionObject through the full 9-stage pipeline (validation → RTQL → value → trust → authority → certificates → 7-gate → state machine → priority) and return verdict + certificate chain + audit trail.
- **Why:** Framework 5.4 (Decision Taxonomy D0–D6) + 5.5 (QC→VC→TC→EC certificate chain) + 5.6 (7-Gate Authorization) + 5.1 (RTQL) — this is *the* canonical decision surface; codifying it as MCP makes "should I do X?" a first-class verb available to every Claude session and to external consumers via gateway. **Human super-ability:** a single operator runs the governance check a 5-person review committee previously ran (Gigent Value Matrix audit chain hit in full).
- **Where:** `decision-engine/api/routes.py:163` (`POST /v1/decisions/process`); already gateway-routed at `/v1/decisions/` prefix. Gap to MCP-ready: thin wrapper + Pydantic schema mirror; no new logic.
- **When:** **Python-stable**. Endpoint shipping since v2.0.0; CLAUDE.md cites it as the primary endpoint. Codification readiness: **HIGH — ship in iteration 02 deploy window**.
- **How:**
  - **Input schema:** `FullDecisionRequest` (already defined at `api/schemas.py`) — 22+ field DecisionObject with `decision_class`, `value_scores` (12-dim), `trust_scores` (7-dim), `alignment_scores`, optional `rtql_input`, `evidence_refs`, etc.
  - **Output schema:** `PipelineResponse` — `verdict` (execute|recommend|escalate|block|needs_data), `certificate_chain` (QC/VC/TC/EC statuses), `audit_log[]`, `executive_summary`, `next_state`, `priority_score`.
  - **Provider abstraction:** N/A at endpoint boundary (pipeline is deterministic); any LLM calls inside route through `_invoke_llm()` (remediated 2026-05-05 per CLAUDE.md).
  - **Audit envelope:** Pipeline already emits per-stage audit entries via `engine/audit.py:serialize_pipeline_result`. MCP wrapper adds `tool_name=gigaton.decision.process`, `tool_version=1.0.0`, `schema_version=2.0.0`, `caller_session_id`.
- **Context:**
  - **Dependencies:** decision-engine Cloud Run service must be reachable (it is — `DECISION_ENGINE_URL` already an env var in gateway).
  - **Consumers:** every other engine that wants governed action (PDC Phase 4 capability writes, sales-operating-system agent deployments, PPEME scenario writes, SIE operator approvals).
  - **Risk if exposed prematurely:** LOW — the pipeline is the gatekeeper, so wrapping it makes the *correct* path easier; risk is opposite (bypassing it). Iteration 02 should propose a drift-sentinel rule penalizing direct engine calls that should route through `decision.process`.

### 2. `gigaton.decision.transition`

- **What:** Advance a decision's state in the lifecycle state machine (`draft → certified → executing → reviewed → outcome_recorded`) with validity check.
- **Why:** Framework 5.4 (Decision Taxonomy lifecycle) + 5.7 (Learning Loop — outcome-recording is the loop's closing edge). **Super-ability:** an operator drives a decision through its full audited lifecycle without touching SQL or manually mutating state. The state machine is the human-readable expression of governance.
- **Where:** `decision-engine/api/routes.py` (`POST /v1/decisions/transition`); already gateway-routed.
- **When:** **Python-stable**; state machine valid transitions live in `engine/state_machine.py` and config.
- **How:**
  - **Input schema:** `TransitionRequest` — `decision_id`, `target_state`, `actor`, optional `reason`, `evidence_refs[]`.
  - **Output schema:** `TransitionResponse` — `success`, `current_state`, `target_state`, `blocking_reason?`, `audit_entry_id`.
  - **Provider abstraction:** N/A.
  - **Audit envelope:** Every transition writes to audit table; MCP wrapper carries `caller_session_id` so we attribute the Claude session that advanced the state.
- **Context:**
  - **Dependencies:** decision must exist; reachable via `gigaton.decision.process` first.
  - **Consumers:** SIE operator approvals (they transition `certified → executing` when the operator clicks approve), PDC affiliate workflow.
  - **Risk:** LOW — the state machine rejects invalid transitions; MCP wrapper inherits that safety.

### 3. `gigaton.outcome.record`

- **What:** Record actual outcome for a decision; closes the learning loop and feeds the 4 calibration engines (OVS, RTQL, ROI, value).
- **Why:** Framework 5.7 (Learning Loop + 4 Calibration Engines) — this is the *single most underused* high-leverage verb in the ecosystem. Today most Claude sessions record outcomes ad-hoc in markdown; codification means every recorded outcome trains the platform. **Super-ability:** an operator's daily verdict ("this worked / didn't") becomes calibration weight for the next decision the *whole platform* makes.
- **Where:** `decision-engine/api/routes.py` (`POST /v1/outcomes/record`); already gateway-routed.
- **When:** **Hybrid → Python-stable**. The endpoint is stable; the human entry of outcomes is still ad-hoc — codifying the MCP verb is what *creates* the codification flywheel here.
- **How:**
  - **Input schema:** `OutcomeRequest` — `decision_id`, `outcome_class` (executed|abandoned|partially_executed), `actual_value_delta`, `unexpected_costs`, `surprise_score` (0–1), `human_override_reason?` (feeds Human Override Engine — see backlog), `recorded_by`, `recorded_at`.
  - **Output schema:** `{ recorded_id, learning_summary_link, calibration_deltas: { ovs: ..., rtql: ..., roi: ..., value: ... } }` — show the operator what their honesty just trained.
  - **Provider abstraction:** N/A at boundary; any calibration LLM call inside is centralized.
  - **Audit envelope:** Already part of decision-engine audit log; MCP adds `tool_name`, `tool_version`, `caller_session_id`.
- **Context:**
  - **Dependencies:** the matching decision must exist in `drift_history.db` / decision store.
  - **Consumers:** OVS-Calibration Engine (when promoted from backlog), Human Override Engine (3× outcome weight when override flag is set), Sunday weekly initiative report.
  - **Risk:** LOW. Recording a false outcome poisons calibration — but the surface is bounded to attributed operators and audit-trailed.

## Cross-cutting observations

1. **Decision-engine is the gravitational center.** Of the 10 candidates, four (`decision.process`, `decision.transition`, `outcome.record`, `drift.scan`) live in decision-engine or its drift_sentinel subtree, and two more (`access.check`, `signoff.required`) feed gates inside decision-engine. **Recommendation:** the *first* gigMCP server to ship should be `gigaton-mcp-decision` covering these 4 + an explicit feed for the next 2. Single repo, single deploy, single audit boundary.

2. **Gateway prefixes are already the canonical namespace.** Every gateway-routed prefix in `routing_table.py` maps cleanly to a `gigaton.<engine>.<verb>` MCP tool name. The path-prefix → engine table is the *de facto* gigMCP catalog; we should formalize it. **Recommendation:** keep MCP tool names 1:1 with gateway path-prefix tail; reject candidates that don't have a gateway entry until they do.

3. **The scaffold engines (UAE, HME, PPEME) are 1 deploy away from contributing 4–6 more candidates each.** None of `state.estimate`, `forecast.simulate`, `events.ingest`, `coaching.get`, `analysis.latest`, `task.next` made the iteration 01 cut because their endpoints are declared but not stably shipped. **Iteration 03 should re-evaluate after Monday Post-Deploy Monitoring Runbook's T+72hr checkpoint clears for these engines.**

4. **The outcome-recording verb is the codification flywheel's bottleneck.** Today, sessions record outcomes in markdown memory files (e.g., session handoffs, audit memos), not via `POST /v1/outcomes/record`. Until `gigaton.outcome.record` is an MCP tool and Claude sessions are *trained to call it* (codify the trigger), the learning loop's closing edge stays open. **This is the single highest-leverage candidate measured by future-marginal-cost reduction.**

5. **B-05 (pricing decisions bypass decision-engine governance) is the most expensive open gap.** `pricing.calculate` shipping as MCP without the decision-engine bridge institutionalizes the bypass. **Recommendation:** ship `decision.process` and `outcome.record` MCP first; ship `pricing.calculate` MCP only after the bridge (or after the MCP wrapper internally emits a decision shadow record).

6. **No candidate this iteration surfaces a provider-lock-in or prompt-version drift risk.** The 10 candidates are either deterministic (drift, access, signoff, recommendation, pricing math) or pass through LLM calls that have *already* been remediated to use `_invoke_llm()`. This is a healthy signal — the codification candidates are the *stable* surface, by definition. Iteration 02 should re-audit `intelligence.query` and the pricing reasoning chain to verify this holds at the implementation level (not just the architectural level).

7. **The directive's "rule promotion at >10× / 3 weeks" mirror applies here.** The candidates above are sized to fit the window: 10 candidates × 7 iterations ≈ 70 codification opportunities by 2026-05-27. Realistic ship count: 6–8 MCP tools shipped to the `gigaton-mcp-*` servers by end-of-window, with the remainder graduating in the post-window codification cycle.
